Topic 1

1.1

This topic focuses on the client, customer, and development side of programming. This includes documentation, change management, system compatibility, context when developing the system, data migration, testing, data loss, and software deployment.

1.1.1

System means a program or a piece of software.

The context when developing a new system is a key factor as to what and how the system will function. This can include:
1. How the system is being developed?
    a. Is it one developer or a whole team?
    b. Which programming language is it going to use? On what OS will it run?
    c. How can users download and use it, if they are able to?
2. Why is it being developed?
3. What is being developed? What are the requirements?
4. Who is asking for the system? Do they have specific requirements?
5. When and where is it being developed?

Two aspects that are important are the limitations and the extent of the system. How will users be affected? If a team is developing the new system, how will that team have to change or adapt to fit the new workflow? Are there going to be limitations on the technology when developing the new system? There are many more questions that you will be able to ask about the new system that can help understand the context of the development of the system better.

This is mostly used for the Computer Science IA during the planning and development stages.

1.1.2

Changing a system can take a lot of money, time, and people to change or update.

There are multiple disadvantages to changing a system:
1. Users may dislike the changes made
2. The new system might not have all the same features as the old system
3. The older system might have been faster
4. There might be a compatibility issue with the new system
5. It's expensive to create the new system and remove the old one
6. There can be data loss during the transfer of data

There are also many advantages of changing to a newer system which aren't listed here. Go to the Resources page to find useful links which might have them.
